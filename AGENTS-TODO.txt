Why this work exists
--------------------
Every serious AI agent harness shipping in 2026 — coding agents,
research agents, ops agents — is converging on the same architecture:
deterministic state machine on the outside, agent reasoning on the
inside. Phase transitions (requirements → architecture → tasks →
implementation), tool-scope gating per phase, critic loops with
bounded retries, human-in-the-loop approvals over Slack / email /
chat, durable resume after worker crashes, token + USD budget caps.

That outer layer is workflow-engine work, not agent work. Today most
teams either bolt it on with ad-hoc scripts or pull in a heavyweight
framework (LangGraph, Temporal) that drags in vendor abstractions
their agent code does not actually need.

Hippo is the right shape for this layer — Postgres-native, durable,
TypeScript-first, light operationally — but it does not yet have the
generic primitives that make agent-shaped workloads first-class.
This roadmap closes that gap WITHOUT turning Hippo into an
agent-only tool.

Positioning rules
-----------------
- Hippo stays a generic Postgres durable workflow engine. NOT an
  "agent engine". NOT Hermes-specific. NOT Claude-specific.
- Every primitive added here must have at least one credible
  non-agent use case (long-running ML training jobs, browser
  automation, video transcode, batch pipelines, webhook-driven
  async work, payment settlement). If a feature is agent-only, it
  goes in a separate `@hippo-contrib/agents` package, not core.
- Goal: make Hippo the best Postgres workflow engine for long-
  running external sessions with metered usage and live event
  streams. Agents fall out for free as one consumer of that.
- README pitches the agent angle alongside the traditional one so
  agent builders find Hippo, but core stays generic so non-agent
  consumers are not forced to ignore agent vocabulary.

Motivating consumers (out-of-tree)
----------------------------------
Modern agent harnesses and long-running automation systems need phase
transitions, HITL gates, durable retries with critic feedback, operator
rewind when external work goes sideways, usage budgets, and live event
streams into dashboards. All of those map to generic primitives below.
Out-of-tree integrations prove the primitives on real workloads while
core stays focused on the workflow engine contract.

Design rules
------------
1. Core stays language-agnostic at the boundary. External sessions
   are addressed by opaque string IDs; Hippo never imports an SDK for
   the external system.
2. Every primitive must compose with existing store semantics:
   same-txn step commit, lease + recovery, rewind/fork, child runs,
   compensation, partitioned history.
3. No agent-specific verbs in public types. `agentStep` lives in
   contrib; core ships `externalSession`, `usage`, `events`, etc.
4. Every Tier 1 item must ship with: SQL migration + store method +
   engine wiring + SDK builder + integration test + dashboard pass.

Tier 1 — Required for credible "long-running external session" story
====================================================================

[x] 1. externalSession step kind
    Generic problem: an external system holds state for the step
    (agent session, ML training job, browser automation context,
    Stripe payment intent). Step must persist the external id so
    that after worker crash/recovery we can REATTACH instead of
    restarting the external work.

    API sketch:
      externalSession({
        kind: "string-tag",          // free-form taxonomy, e.g. "agent", "ml-train"
        start: async (ctx) => ({ externalId, initial }),
        poll:  async (ctx, externalId) => ({ status, output? }),
        attach?: async (ctx, externalId) => ({ status, output? }),
        cancel?: async (ctx, externalId) => void,
        timeoutMs?: number,
      })

    Persistence: new column workflow_step_attempts.external_session_id
    (text, nullable, indexed). Plus optional external_session_meta
    (jsonb) for kind/url/labels visible in dashboard.

    Recovery: if a step attempt is recovered mid-flight and the
    attached external_session_id is non-null, engine calls attach()
    instead of start(). attach() defaults to poll() if not provided.

    Non-agent example: video transcoder API. start() POSTs job, gets
    transcode_id; poll() returns status; attach() resumes polling
    after worker recovery.

[ ] 2. External heartbeat protocol
    Generic problem: step is alive but waiting on an external system;
    lease shouldn't expire just because we haven't ticked locally.

    Existing primitive (ctx.heartbeat) updates lease_expires_at and
    last_heartbeat_at. Extend to accept inbound heartbeats from the
    external system via signed callback:

      POST /v1/steps/:attemptId/heartbeat
        Authorization: Hippo callback HMAC
        body: { progress?: number, message?: string, usage?: {...} }

    Side effects: extend lease, append a workflow_event of kind
    "step.heartbeat" with provided progress/message, optionally
    record usage (see Tier 1.4).

    Non-agent example: long S3 multipart upload posts progress
    heartbeats so workers don't expire its lease.

[ ] 3. Step event streaming (typed event bus)
    Generic problem: step emits intermediate events while running.
    Consumers (dashboards, monitoring) want a live tail without
    waiting for step completion.

    Today: workflow_events table exists, SSE infrastructure exists,
    but step bodies have no typed channel to emit progress events
    by step kind.

    API:
      ctx.emit({ type: "tokens", data: { delta: "..." } })
      ctx.emit({ type: "tool-call", data: { name, args } })
      ctx.emit({ type: "progress", data: { pct: 0.42 } })

    Event types are open strings. Hippo persists them in
    workflow_events with step_attempt_id + event_type. SSE channel
    surfaces them live.

    Non-agent example: build pipeline step emits "log-line", "test-
    result", "artifact-uploaded" events.

[ ] 4. Usage ledger + budget caps
    Generic problem: steps consume metered resources (tokens, bytes,
    GPU-minutes, third-party API quota). Workflow should cap total
    consumption and dashboard should aggregate.

    New table: workflow_run_usage
      run_id uuid, step_attempt_id uuid?, resource text,
      amount numeric, cost_usd numeric, recorded_at timestamptz
    Hash-partitioned by run_id consistent with existing history.

    API:
      ctx.recordUsage({ resource: "tokens", amount: 1234,
                        costUsd: 0.0089, dimension?: "input" | "output" })

    Workflow declaration:
      defineWorkflow({ ..., budget: { tokens: 2_000_000,
                                      costUsd: 5.00 } })

    Enforcement: on usage record, sum-check budgets; on exceed,
    transition run to terminal state `exhausted_budget` (NEW). First-
    class state, not a generic error.

    Non-agent example: S3 egress costs per step; per-workflow USD cap
    prevents runaway data export jobs.

[ ] 5. External cancellation hooks
    Generic problem: Hippo cancels a step (hard terminate or budget
    exhaustion). External system holding the session needs to know to
    abort its work. Compensation hooks cover COMPLETED steps; this is
    the IN-FLIGHT case.

    API: externalSession step accepts:
      cancel: async (ctx, externalId) => void

    Hard terminate path: before transitioning step state, engine
    calls cancel() if defined. cancel() is best-effort, capped by
    timeout, failures logged but do not block the cancel itself.

    Non-agent example: cancel a long-running HTTP request to a
    third-party API when the workflow is killed.

Tier 2 — Generic ergonomics that benefit any long-running step
==============================================================

[ ] 6. Per-run scoped key-value side channel
    Generic problem: steps need to stash small bits of state across
    retries / phases without bloating the payload.

    Today: only payload + step output exist. Payload is conceptually
    input; output is per-attempt. Neither is the right shape for a
    cross-step scratchpad.

    New table: workflow_run_kv (run_id, key, value jsonb, updated_at).
    API: run.kv.get(key), run.kv.set(key, value), run.kv.delete(key).
    Scoped to the run only.

    Non-agent example: stash an API access token refreshed by one
    step, used by subsequent steps without re-fetching.

[ ] 7. Step taxonomy + dashboard renderer plugin API
    Generic problem: dashboard renders all steps with the same
    generic UI. Different step kinds (agent, build, payment, browser
    automation) benefit from rich per-kind renderers.

    API: workflow registration includes step kind tags. Dashboard
    accepts plugin bundles that register renderers per kind:
      registerStepRenderer("agent", AgentPanel)
      registerStepRenderer("payment", PaymentPanel)

    Hippo ships generic renderer for unknown kinds. Renderers are
    pure functions of step state + events; no privileged dashboard
    access.

    Non-agent example: payment step shows Stripe charge state +
    refund button; build step shows test pass/fail timeline.

[ ] 8. Retry-with-augmented-input pattern
    Generic problem: retry where each attempt should be informed by
    the previous attempt's output + failure reason. Today retries are
    stateless re-calls.

    API:
      iterativeStep({
        initial: { ... },
        refine: (prevInput, prevOutput, error) => nextInput,
        attempt: async (ctx, input) => output,
        maxAttempts: 3,
      })

    Critic loop is one consumer; constraint solvers and adaptive
    rate-limiting are others.

    Non-agent example: API retry where prior 429 response headers
    adjust the next attempt's request rate.

[ ] 9. Operator action plugin API
    Generic problem: dashboard needs custom operator actions per
    step kind beyond cancel/retry/rewind.

    API: workflow declares operator actions per step kind:
      registerOperatorAction("agent", "inject", async (ctx, args) => ...)
      registerOperatorAction("payment", "refund", async (ctx, args) => ...)

    Hippo provides dispatch + audit log row in workflow_events with
    kind="operator.action". Consumer provides the semantic handler.

    Non-agent example: payment dispute resolution; manual override
    on any human-in-loop step.

Design notes: externalSession MVP (PR 1)
========================================

Scope kept tight so first PR lands and reviews. Polling, attach(),
cancellation propagation, and budget integration all land in
follow-up PRs.

MVP step kind shape
-------------------
  type ExternalSessionStartResult = {
    externalId: string         // opaque to Hippo
    payload?: JsonValue        // optional initial state stored on wait
  }

  type ExternalSessionStepDefinition = {
    kind: "externalSession"
    label?: string
    sessionKind: string        // free-form tag e.g. "agent", "ml-job"
    next?: string
    transitions?: Record<string, string>
    timeoutMs: number          // hard deadline for the whole session
    retry?: RetryPolicy        // applied to start() failures only
    start: (ctx) => Promise<ExternalSessionStartResult>
    resume: (ctx, externalId, payload?) => Promise<WaitStepResumeResult>
  }

Engine flow
-----------
1. Step starts: call start(ctx) — gets { externalId, payload? }.
2. Engine opens a wait record with:
     correlationKey       = `ext:${sessionKind}:${externalId}`
     externalSessionId    = externalId
     externalSessionKind  = sessionKind
     expiresAt            = now + timeoutMs
3. Step transitions to waiting; same lifecycle as existing wait.
4. External system completes work, POSTs callback (route below).
5. Engine resolves wait by externalSessionId (or correlationKey),
   calls resume(ctx, externalId, payload).
6. Result transitions step like any other.

Storage
-------
Reuse wait infrastructure rather than build a parallel table. New
columns:

  ALTER TABLE workflow_step_attempts
    ADD COLUMN external_session_id text,
    ADD COLUMN external_session_kind text;

  ALTER TABLE workflow_waits
    ADD COLUMN external_session_id text,
    ADD COLUMN external_session_kind text;

  CREATE INDEX workflow_waits_external_session_idx
    ON workflow_waits (external_session_id)
    WHERE external_session_id IS NOT NULL;

  CREATE INDEX workflow_step_attempts_external_session_idx
    ON workflow_step_attempts (external_session_id)
    WHERE external_session_id IS NOT NULL;

Callback route
--------------
  POST /v1/external-sessions/:externalId/resume
    Authorization: Hippo callback HMAC (reuses verifyCallbackRequest)
    body: { payload?: JsonValue }

  Lookup: workflow_waits WHERE external_session_id = :externalId
          AND status = 'open'. If found, dispatch to existing
          resumeWait engine path; otherwise 404.

  Convenience: correlationKey route still works
  (POST /v1/waits/:correlationKey/resume) for callers that prefer
  to compute the key themselves.

Recovery semantics (MVP)
------------------------
Step lives inside a wait. Existing wait recovery covers the
"worker died between start() and openWait()" case via lease
expiry. The interesting REATTACH semantics (call attach() on
recovery with persisted externalId) land in PR 2.

For the MVP, recovery behavior is: if start() succeeded and the
wait is open, the step is "safe" — it will resume on callback or
expire on timeout. If start() failed mid-flight (external work
might have been kicked off but we didn't persist the id), retry
policy on start() applies; the external system either ignores the
duplicate kickoff or returns an idempotency-keyed existing id.
Callers handle externalSession start() idempotency themselves for
PR 1 (documented).

Example workflow (non-agent)
----------------------------
examples/workflows-demo/src/workflows/video-transcode.ts

  Step 1: queueTranscode (externalSession)
    sessionKind: "video-transcode"
    start: POST to fake transcoder service → { externalId }
    resume: validate transcode output, store result url in patch

  Step 2: notifyDone (task)
    output url logged

  Demonstrates: long-running external work, callback-driven
  resumption, no agent concept anywhere.

Open decisions (need sign-off before code)
-------------------------------------------
A) Reuse wait table vs new external_sessions table
   Decision: REUSE. Less duplication, recovery / cleanup paths
   already exist. Trade: schema knows externalSession kind exists.
   Acceptable — wait table already knows about signal kind.

B) sessionKind tag — free string or enum?
   Decision: FREE STRING. Open by design; closing it would force
   contrib packages to register kinds with core. Dashboard renders
   unknown kinds as generic.

C) Callback auth — HMAC only or token?
   Decision: HMAC. Reuse existing callback authenticator. Token
   auth on the start side (via app api token) is fine because
   start is initiated by Hippo itself; the callback is the
   externally-driven path that needs HMAC.

D) timeoutMs required?
   Decision: YES, required. Matches wait/signal. Prevents zombie
   external sessions. Long-running consumers pick a high number;
   making it optional is a footgun.

E) retry policy on start() but not resume()?
   Decision: YES. start() is the side-effect to the external
   system; resume() is internal book-keeping. resume() errors
   should fail the step attempt, not silently retry.

F) PR 1 ships without poll / attach / cancel?
   Decision: YES, those are PR 2/3/4. MVP is callback-driven only.

Design notes: external heartbeat protocol (PR 2)
================================================

Builds on PR 1. Extends ctx.heartbeat (already exists for owner-side
heartbeats) with an externally-driven counterpart so long-running
external sessions can extend their lease and report progress without
needing the worker process to tick.

API
---
  POST /v1/external-sessions/:externalId/heartbeat
    Authorization: Hippo callback HMAC
    body: {
      progress?: number      // 0..1
      message?: string
      usage?: {              // matches recordUsage shape (PR 4)
        resource: string
        amount: number
        costUsd?: number
      }
    }

Effects (all in one txn):
  - workflow_runs.lease_expires_at extended by configurable grace
    (default: matches existing lease_ms)
  - workflow_step_attempts.last_heartbeat_at = now()
  - workflow_events row of type "step.external_heartbeat" with body
  - if usage provided AND PR 4 has landed, recordUsage is called

Open decisions
--------------
A) Heartbeat resets lease vs adds fixed grace?
   Leaning: RESETS to now + lease_ms. Matches existing semantics.
B) Heartbeat for non-externalSession steps?
   Leaning: NO. The internal ctx.heartbeat already covers task steps.
   This endpoint exists specifically for the external-side caller.
C) Rate limit?
   Defer. Honest rate-limiting needs a separate primitive. For PR 2
   document expected cadence (every 5–30s) and rely on operator
   discipline.

Design notes: step event streaming (PR 3)
=========================================

Generic: any step body can emit typed intermediate events that pipe
into the existing SSE event tail. Non-blocking, fire-and-forget from
the step body's perspective.

API
---
  ctx.emit({ type: string, data: JsonValue }) => Promise<void>

Persistence:
  workflow_events row with event_type = "step.emit:<type>" and a
  payload that includes { type, data, stepKey, stepAttemptId }.

For externalSession steps, the callback heartbeat (PR 2) can carry
events too:
  POST /v1/external-sessions/:externalId/events
    body: { events: [{ type, data }, ...] }
  Same persistence shape.

Dashboard:
  SSE tail already streams workflow_events; add a per-step filter so
  the run detail view shows emit events grouped under their step
  attempt. Step kind renderers (PR 7) can render these richly.

Open decisions
--------------
A) Event types validated by Hippo or open strings?
   Leaning: OPEN STRINGS. Step kind renderers handle unknown types.
B) Bulk emit (batch into one row vs row-per-event)?
   Leaning: ROW PER EVENT for now. Partition + index already handle
   volume. Revisit if a single step emits thousands.
C) Retention?
   Same retention rules as existing workflow_events. Archival is a
   separate concern owned by the existing partitioning work.

Design notes: usage ledger + budget caps (PR 4)
===============================================

Generic: any step can record metered usage. Workflow declares budgets;
ledger sums on insert and transitions the run to a new terminal state
when exceeded.

API
---
  ctx.recordUsage({
    resource: string,             // free string e.g. "tokens", "bytes"
    amount: number,
    costUsd?: number,
    dimension?: string,           // optional sub-tag e.g. "input"|"output"
  })

  defineWorkflow({
    ...,
    budget?: {
      [resource: string]: number  // amount cap per resource
      costUsd?: number            // hard USD cap across all resources
    }
  })

Storage
-------
New table workflow_run_usage, hash-partitioned by run_id to match the
existing history tables:

  CREATE TABLE workflow_run_usage (
    id BIGSERIAL,
    run_id UUID NOT NULL,
    step_attempt_id UUID,
    resource TEXT NOT NULL,
    dimension TEXT,
    amount NUMERIC NOT NULL,
    cost_usd NUMERIC,
    recorded_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (run_id, id)
  ) PARTITION BY HASH (run_id);

Plus run-level rollup column on workflow_runs:
  ALTER TABLE workflow_runs
    ADD COLUMN total_cost_usd NUMERIC NOT NULL DEFAULT 0;

Enforcement
-----------
On recordUsage insert (same txn as step commit when transactional):
  1. Insert into workflow_run_usage
  2. Update workflow_runs.total_cost_usd += costUsd
  3. Sum amount-per-resource for the run, compare to budget
  4. If any cap exceeded, transition run status to "exhausted_budget"
     (NEW terminal state — see types changes)

The transition happens at the END of the step (in the same txn that
records the usage), so a step that records usage above the cap still
completes its work — the workflow does not progress past it.

New terminal state
------------------
WorkflowRunStatus gains "exhausted_budget". Compensation hooks fire
on this state like other terminal failures.

Open decisions
--------------
A) Cap behavior: stop immediately vs let current step finish?
   Decision: LET CURRENT STEP FINISH. Killing mid-step is destructive
   and we already pay for the work. Workflow does not advance past it.
B) Per-step vs per-workflow budget?
   Leaning: PER-WORKFLOW first. Per-step is a follow-up if asked.
C) Currency?
   USD only in core. Other currencies are FX layer concerns.

Design notes: external cancellation hooks (PR 5)
================================================

Completes the externalSession story. When Hippo cancels a step (hard
terminate, budget exhaustion, parent run cancel), the external system
needs to know to abort. Compensation hooks cover COMPLETED steps;
this covers IN-FLIGHT.

API
---
  ExternalSessionStepDefinition gains:
    cancel?: (
      context: StepExecutionContext,
      externalId: string,
      reason: "hard_terminate" | "budget" | "parent_cancel"
    ) => Promise<void> | void

Engine
------
On hard terminate path, before transitioning run to canceled:
  - For each open externalSession wait on the run, call cancel()
  - cancel() runs with a strict timeout (default 5s, configurable)
  - cancel() failures are logged as workflow_events
    "step.external_cancel_failed" but DO NOT block the cancel itself
  - Wait record transitions to "canceled"

Open decisions
--------------
A) cancel() retry?
   Leaning: NO. Best effort, fire once, log on failure. Retries on
   cancel paths add latency where users want fast cancellation.
B) Should the external HTTP API expose a cancel endpoint too?
   Decision: NO. Hippo initiates the cancel; the external system
   reacts. If callers want symmetric "external aborts itself" they
   call POST /v1/runs/:id/cancel from the external side.

PR cadence
==========
Working model: each Tier 1 / Tier 2 item lands as its own PR to main.
The feat/agent-primitives branch is the long-lived parent during the
Tier 1 push, rebased forward as upstream main advances.

Lifecycle per item:
  1. Pick item from working order
  2. Update "Current branch state" with the open work in flight
  3. Implement (migration → types → store → engine → routes → tests
     → example workflow → docs)
  4. Run full test suite + npx tsc --noEmit + npm run lint
  5. Update README "For AI Agents" if the item exposes new primitives
     visible to agent harnesses
  6. Open PR. PR description links to this file's design notes for
     the item plus the open-decisions block.
  7. On merge, flip the Tier 1/2/3 checkbox here. Move design notes
     for landed items into a separate "Landed work" section below
     for archival.

Branch hygiene:
  - Keep AGENTS-TODO.txt the FIRST file in every commit so it always
    reflects current state.
  - Never delete design notes — move them to "Landed work" archive.
  - If a decision in this file is overruled in code review, update
    BOTH the code and the file in the same PR.

Testing strategy
================

Unit tests
  - Pure logic in src/lib/*.test.ts using existing in-memory store
    fixture (see workflow-engine.test.ts mock store pattern).
  - Every new step kind must have engine-level unit tests covering:
    happy path, retry on transient error, timeout, hard terminate.

Postgres-backed integration tests
  - Every new schema change must have an integration test that
    exercises real Postgres (see workflow-store.pg.test.ts).
  - For externalSession: spin up a run, simulate external callback,
    assert wait resolves and step advances.
  - For partitioned tables: assert partition pruning via EXPLAIN

@hippo-contrib/agents (separate repo / package)
===============================================

Out of scope for this branch BUT context for why core primitives are
shaped the way they are. The eventual agent-specific layer:

Shape:
  @hippo-contrib/agents
    src/
      agent-step.ts          — wraps externalSession with agent vocab
      hermes-adapter.ts      — calls Hermes gateway HTTP API
      claude-sdk-adapter.ts  — calls Claude Agent SDK
      openai-adapter.ts      — calls OpenAI Assistants API
      critic-loop.ts         — wraps iterativeStep (Tier 2.8) for critic
      slack-approval.ts      — wraps wait.forSignal with Slack callbacks
      tool-allowlist.ts      — typed tool gating helpers
      usage-helpers.ts       — token-to-cost converters per provider

The contrib package does NOT live in this repo. The Hippo core team
maintains the primitives; the contrib layer is community-extensible.

What this implies for core API design:
  - externalSession takes a free-string sessionKind, not a closed
    enum, so contrib packages can label sessions without core
    needing to know.
  - ctx.recordUsage takes a free-string resource, not "tokens" enum.
  - Step taxonomy is open (Tier 2.7) so contrib renderers register
    without core registry changes.
  - Operator actions are pluggable (Tier 2.9) so contrib UIs ship
    domain-specific buttons.

Glossary
========

externalSession
  A step whose work happens outside the Hippo worker process. The
  external system holds state behind an opaque ID; Hippo waits for
  a callback to advance.

sessionKind
  A free-string tag on externalSession that lets dashboards render
  the step differently and lets contrib packages address sessions
  by category without core needing to know the type.

usage ledger
  Per-run record of metered resource consumption (tokens, bytes,
  GPU-minutes, USD). Aggregated for budget enforcement and
  dashboard cost panels.

iterative step (planned, Tier 2.8)
  A retry pattern where each attempt sees the previous attempt's
  output and failure reason. The critic loop in agent workflows
  is one consumer; constraint solvers and adaptive rate limiters
  are others.

contrib (as in @hippo-contrib/agents)
  Out-of-tree packages that compose Hippo primitives into
  domain-specific layers. Hippo core stays generic; contrib layers
  carry the vocabulary of a particular domain (agents, payments,
  ETL, etc.).
